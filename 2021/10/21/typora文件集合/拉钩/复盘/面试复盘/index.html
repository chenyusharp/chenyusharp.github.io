乐其电商的面试复盘：

乐观锁和悲观锁的区别以及使用场景；

悲观锁：每次获取数据的时候，都会进行加锁，确保自己使用的过程中数据不会被别人修改，使用完进行解锁。由于数据进行加锁，期间对数据的读写的其他线程都会进行等待；所以我们说，乐观锁适合写比较多的场景。如果读的操作比较多，每次的读取都进行加锁的话，就会增加大量的系统开销，降低系统的吞吐量。

乐观锁：每次获取数据的时候都不会进行加锁，但是在更新的数据的时候需要判断该数据是否被别人修改过。如果被修改过，则不进行更新，否则进行更新。所以我们说，乐观锁适合读取操作比较多的场景。如果写的操作比较大，那么数据发生冲突的可能性就会增大，为了保证数据的一致性，应用层需要不断的重新获取数据，这样会降低系统的吞吐量。

总结：

悲观锁比较适合强一致性的场景，但是效率比较低，特别是读的并发性低。乐观锁则适用于读多写少，并发冲突少的场景。

    • ApplicationContext：它作为BeanFactory的子接口，除了提供BeanFactory所具有的功能外，还提供了更完整的框架功能。

  1. 数据库的主键索引为什么采用数据库的自增呢？
  • 业务上

业务主键只是理论上唯一的，实际上有可能重复。所以说设置一个与业务无关的自增Id作为主键。

  • 技术上

如果表使用自增主键，那么每次插入新的记录，记录就会顺序添加到当前索引节点的后续位置，当一页写满以后，就会自动开辟一个新的页。总的来说，可以提高查询和插入的性能。

  • 为什么？

主键索引中既存储索引值，又在叶子节点存储行的数据，数据文件和索引文件是在同一个文件中；

1⃣️如果没有定义主键，则会使用非空的UNIQUE键做主键；如果没有非空的UNIQUE键，则系统生成一个6字节的rowId做主键；

2⃣️聚簇索引中，N行形成一页（16K）。如果碰到不规则数据插入时，为了保持B+树的平衡，会造成频繁的页分裂和页旋转，插入速度比较慢。所以说主键索引的值应尽量是连续增长的值，而不是随机的值。

  1. IO密集型和CPU密集性的线程怎么设置？

​ CPU密集的意思是该任务需要大量的运算，而没有阻塞，CPU一直全速运行。CPU密集型任务配置尽可能少的线程数量：一般公式：CPU核数+1个线程的线程池。为什么？减少上下文的切换。

由于IO密集型任务线程并不是一直在执行任务，则应配置尽可能多的线程，如CPU核数*2。为什么？IO密集型，即该任务需要大量的IO，即大量的阻塞。配置多线程的话，其他的线程可以执行其他的任务。

  1. 什么是伪共享；

    缓存系统中是以缓存行（cache line）为单位存储的。缓存行是2的整数幂个连续的字节，一般是32～256个字节。最常见的缓存行大小是64个字节。当多线程修改相互独立的变量时。如果这些变量共享同一个缓存行，就会无意中影响彼此的性能，这就是伪共享。

  5. concurrentHashMap的jdk 7与jdk 8的区别？

    1.7

    首先将数据分为一段一段的存储，然后给每一段数据分配一把锁，当一个线程占用锁访问其中的一段数据时，其他段的数据也能被其他线程访问；

    concurrentHashMap是有segment数组结构和HashEntry数组结构组成的。

    segment实现了ReentrantLock，所以segment是一种可重入锁，扮演锁的角色。HashEntry用于存储键值对数据。

    一个consurrentHashMap里面包含了一个segment数组。segment的结构和HashMap类似，是一个数组加链表的结构。一个segment包含一个HashEntry数组，每个HashEntry是一个链表结构的元素，每个segment守护着一个HashEntry数组里面的元素，当对HashEntry数组中的数据进行修改时，必须首先获得对应的segment的锁。

    1.8

    ConcurrentHashMap取消了segment分段锁，采用CAS和synchronized来保证并发安全。数据结构跟HashMap 1.8的结构类似，是一个数组+链表/红黑树的结构。

    synchronized只锁定当前链表或红黑树的首节点，这样只要hash不冲突，就不会产生并发。另外，1.6以后，jvm对synchronized的性能进行了优化，所以1.8中的concurrentHashMap的性能比1.7的要高。

  2. spring都用到了哪些设计模式？

    • 工厂模式

      Spring使用工厂模式可以通过 BeanFactory 或 ApplicationContext 创建 bean 对象。两者的区别：BeanFactory是延迟注入的。而ApplicationContext是容器启动的时候，一次性创建所有的Bean。

      ApplicationContext的是三个是实现类：

      1. ClassPathXmlApplicationContext: 把上下文文件当成类路径资源
      2. FileSystemXmlApplicationContext: 从文件系统中的XML文件载入上下文定义信息
      3. XmlWebApplicationContext: 从web系统中的XML文件载入上下文定义信息

    • 单例模式

      一个类只有一个对象。

      优势：

      • 对于频繁使用的对象，可以省略创建对象所花费的时间，这对于那些重量级对象而言，是非常可观的一笔系统开销；
      • 由于 new 操作的次减少，因而对系统内存的使用频率也会降低，这将减轻 GC 压力，缩短 GC 停顿时间。

      spring中的bean的作用域有哪些：singleton、prototype、request、session、global-session。

      Spring 通过 ConcurrentHashMap 实现单例注册表的特殊方式实现单例模式。Spring 实现单例的核心代码如下

    • 代理模式

      具体的应用AOP。AOP是基于动态代理实现的。如果代理的对象实现了某个接口，那么spring采用JDK Proxy去创建代理对象，如果没有实现接口，那么采用Cglib。

      Spring AOP 和AspectJ AOP有什么区别？

      Spring AOP属于运行时增强，而AspectJ AOP是编译时增强。Spring AOP基于代理（动态代理），而AspectJ 基于字节码操作。性能方面，Aspect J 要比Spring AOP要快。

    • 模版方法

      Spring中的JdbTemplate、hibernateTemplate等以Template结尾的对数据库操作的累，使用的是模版模式。

    • 观察者模式

      具体的应用场景：Spring中的事件驱动模型。事件驱动模型中的三个角色：

      • 事件角色：ApplicationContextEvent充当事件的角色。也就是事件的类型。

      Spring中常见的事件：

      1. ContextStartedEvent： ApplicationContext启动后触发的事件；
      2. ContextStoppedEvent: ApplicationContext停止后触发的事件；
      3. ContextRefreshedEvent: ApplicationContext 初始化或刷新后完成后触发的事件；
      4. ContextClosedEvent： ApplicationContext 关闭后触发的事件。

      • 事件的监听者角色：ApplicationListener。

        ApplicationListener是一个接口。里面只定义了一个 onApplicationEvent（）方法来处理ApplicationEvent。我们的实现类中只需要实现这个方法就可以了。titan中的ConsumerHandlerBuilder就是这个逻辑。

      • 时间的发布者： ApplicationEventpublisher

        实际上是通过接口中的publishEvent这个方法实现的。

      总结：

      1. 定义一个事件：实现一个继承自ApplicationEvent，并且写相应的构造函数；

      2. 定义一个事件监听者：实现ApplicationListener，重写里面的onApplicationEvent()

        方法；

      3. 使用事件发布者发布消息：可以通过ApplicationEventPublish的publishEvent（）方法实现。

    • 适配器模式

      spring Aop中的增强或者通知（Advice）使用到了适配器模式，与之相关的接口是AdvisorAdapter。Spring预定义的通知要通过对应的适配器，适配成MethodInterceptor接口类型的对象，然后有MethodInterceptor负责适配Advice。

      spring MVC 中的适配器模式。mvc中，DispatcherServlet根据请求信息调用HandlerMapping，根据请求解析到对应的Handler（Controller控制器），然后由HandlerAdapter适配器处理。找到具体的适配器类用于对目标类进行适配。

    • 装饰者模式

      Spring 中用到的包装器模式在类名上含有 Wrapper或者 Decorator。这些类基本上都是动态地给一个对象添加一些额外的职责。一个应用场景就是动态的修改数据库的数据源。

elasticsearch中的版本号怎么处理的？

es中每一个文档都有一个版本号(_version)，当文档被修改时版本号递增。es使用这个版本号来确保变更以正确的得到执行。如果旧版本在新版本之后到达，它可以被简单的忽略。

我们可以利用_version号来确保应用中相互冲突的变更不会导致数据丢失。我们通过指定想要修改文档的version号来达到这个目的。如果该版本不是当前版本号，我们的请求将会失败。

  • 安全性：具有方法调用/字段粒度的细粒度的白名单。
  • 可选类型：变量和参数可以使用显示类型或动态def类型；
  • 优化：转为Elasticsearch脚本设计。

titan中，采用painless脚本中判断对应的doc中相应部分的文档的lastVer是否小于params中的lastVer的大小，小于的话则更新。同时由于每一个update都是更新整个doc的一部分，如果同时有多个操作更改整个doc的不同部分，这就会导致整个文档的_version版本号出现冲突。（自己的理解，每一次的update的时候，es先是读取对应的文档信息，包括版本号，然后进行值替换，最后进行update。那么这个时候并发的时候就会出现之前获取到的_version版本号被其他的修改给改掉了，就会出现version_conflict_engine_exception这样的异常提示。）所以，在最外部会去捕获这个异常信息，进行一定次数的重试，这样可以最大限度的保证，其他部分的更新也能成功（因为重试的过程中，会重新获取先前递增的版本号）。

平滑扩容：将数据库数量扩容成原来的2倍。

步骤：

  1. 数据库成倍扩容，比如有2个库扩容到4个库；

  2. 配置双主进行数据同步（先测试，后上线）；

  3. ![image-20220622143811444](/Users/xiazhenyu/Library/Application Support/typora-user-images/image-20220622143811444.png)

  4. 数据同步完成以后，配置双主双写（因为同步有延迟，如果时时刻刻都有写和更新操作，会存在不准确的问题）。

![image-20220622143847841](/Users/xiazhenyu/Library/Application Support/typora-user-images/image-20220622143847841.png)

  1. 数据同步完成以后，删除双主同步，修改数据库配置，并重启；

![image-20220622143910092](/Users/xiazhenyu/Library/Application Support/typora-user-images/image-20220622143910092.png)

  1. 此时已经扩容完成，但此时的数据并没有减少，新增的数据库跟旧的数据库一样多的数据，此时还需要写一个程序，清空数据库中多余的数据，如：

    User1去除 uid % 4 = 2的数据;

    User3去除 uid % 4 = 0的数据;

    User2去除 uid % 4 = 3的数据;

    User4去除 uid % 4 = 1的数据;

​ 平滑扩容方案能实现n库扩2n库的平滑扩容，增加数据库服务能力，降低单库一半的数据量。其 核心原理是：成倍扩容，避免数据迁移。

​ 优点：

  • 扩容期间，服务正常进行，保证高可用；

  • 相对停机扩容，时间长，项目组压力没有那么大，出错率低；

  • 扩容期间遇到问题，随时解决，不怕影响线上服务；

  • 可以将每个数据库数据量减少一半。

    缺点：

    • 程序复杂、配置双主同步、双主双写、检测数据同步等；
    • 后期数据库扩容，比如成千上万，代价比较高。

    适用场景：

    • 大型网站
    • 对高可靠性要求高的服务
